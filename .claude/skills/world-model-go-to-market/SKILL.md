# World Model Go-to-Market Strategy

Produce a multi-surface go-to-market plan for world model technology that coordinates consumer acquisition, developer-led growth, and enterprise partnerships into a single coherent strategy. The output is a GTM blueprint with positioning, channel strategy, pricing architecture, vertical penetration sequencing, and competitive response playbooks — all tailored to the unique challenge of bringing a brand-new AI category to market.

## Core Principles

### 1. The Whole Product Closes the Chasm

> "The fundamental basis for marketing in a high-tech enterprise is the development of a whole product model and the discovery of high-value target market segments."
> — Geoffrey Moore, *Crossing the Chasm* ([Goodreads](https://www.goodreads.com/book/show/61329.Crossing_the_Chasm))

World models are a discontinuous innovation. Early adopters will tolerate generating a world, downloading an SPZ file, and manually loading it into a renderer. The early majority will not. A world model GTM must define the "whole product" for each vertical: for robotics, that means generation plus Isaac Sim integration plus collision mesh export plus batch scripting. For architecture, it means generation plus browser-based exploration plus shareable links plus lighting control. Shipping the API alone leaves you stranded in the chasm with a technology demo that only enthusiasts love.

### 2. Positioning Is a Deliberate Act, Not a Tagline

> "Positioning is the act of deliberately defining how you are the best at something that a defined market cares a lot about."
> — April Dunford, *Obviously Awesome* ([Goodreads](https://www.goodreads.com/book/show/45166937-obviously-awesome))

"World model" is not yet a category buyers recognize. The GTM must create the category frame before filling it. This means choosing a competitive reference point that makes the value immediately legible: for developers, the reference is "3D generation API" (analogous to image generation APIs they already use); for enterprise, the reference is "synthetic environment platform" (analogous to simulation tools they already budget for). The positioning must be deliberately different for each surface — Marble App positions against creative tools, the World API positions against infrastructure, and enterprise partnerships position against bespoke simulation pipelines.

### 3. Spatial Intelligence Is the Frontier Beyond Language

> "Spatial intelligence will transform how we create and interact with real and virtual worlds — revolutionizing storytelling, creativity, robotics, scientific discovery, and beyond."
> — Fei-Fei Li, "From Words to Worlds: Spatial Intelligence is AI's Next Frontier" ([Substack](https://drfeifei.substack.com/p/from-words-to-worlds-spatial-intelligence))

The founder's vision is not incidental to GTM — it is the GTM. Fei-Fei Li's framing of spatial intelligence as the successor to language AI gives World Labs a narrative advantage that no competitor can replicate. Every GTM motion should reinforce this frame: we are not a "3D generation startup," we are the company building the spatial intelligence layer. This positions the technology as foundational infrastructure rather than a feature, which justifies platform pricing, attracts long-term enterprise commitments, and creates a moat in developer mindshare.

### 4. Sell the Transformation, Not the Software

> "What we are selling is not the software product — the set of all the features, in their specific implementation — because there are just not many buyers for this software product."
> — Stewart Butterfield, "We Don't Sell Saddles Here" ([nfx.com](https://www.nfx.com/post/slack-memo))

Nobody wakes up wanting to buy "world generation credits." Robotics engineers want to train robots in 1,000 diverse environments without building each one by hand. Game designers want to prototype a level in 30 seconds instead of 3 weeks. Architects want to walk a client through a space that does not yet exist. The GTM must lead with the transformation each vertical experiences — the 100x speedup, the previously impossible workflow, the creative unlock — not with model architecture or credit pricing. Every landing page, demo, and sales deck should answer: "What can you do now that you could not do before?"

### 5. Content 3.0 Is Participatory, Not Passive

> "Content 3.0 is generated by you or together with you."
> — Christoph Lassner, "What if you could talk to your favorite character in a movie?" TED Talk ([TED.com](https://www.ted.com/talks/christoph_lassner_what_if_you_could_talk_to_your_favorite_character_in_a_movie))

Lassner's "Content 3.0" framework redefines the buyer: the consumer is also the creator. This has direct GTM implications. The Marble App is not a passive viewer — it is a creation tool. Acquisition loops should be designed around creation and sharing, not consumption. When a user generates a world and shares a link, every viewer is a potential creator. This is the viral primitive that consumer GTM must be built around, and it differentiates World Labs from competitors like Genie 3 where worlds are ephemeral and cannot be shared.

### 6. Community Is the Moat, Not Code

> "If code isn't a competitive moat, what is? Community. Your open source community is a developer-driven top-of-funnel activity."
> — Peter Levine, "Open Source: From Community to Commercialization" ([a16z.com](https://a16z.com/open-source-from-community-to-commercialization/))

The API examples are open-source. The competitive moat is not in code — it is in the community of developers who build with the API, file issues, create tutorials, and ship integrations. The developer GTM must invest in community infrastructure (Discord, GitHub Discussions, showcase galleries) at least as heavily as in SDK features. Every community-contributed cookbook recipe, every third-party integration, and every Stack Overflow answer compounds into a switching cost that no competitor can replicate by cloning a repository.

### 7. Simulation Will Be the Largest Consumer of Synthetic Worlds

> "Everything that moves will be robotic. We built Omniverse so that everything can have a digital twin."
> — Jensen Huang, GTC Paris Keynote and 3DEXPERIENCE World ([NVIDIA Blog](https://blogs.nvidia.com/blog/huang-3dexperience-2026/))

Huang's vision implies an enormous TAM for synthetic environments — every robot, every autonomous vehicle, every warehouse automation system needs diverse 3D training environments. The World Labs / NVIDIA Isaac Sim integration is not a partnership press release; it is a GTM wedge into a market that will consume millions of worlds per year. The robotics vertical may have a smaller initial user count than gaming, but each customer's volume (batch generation of hundreds or thousands of environments) makes it the highest-revenue vertical. GTM sequencing must account for this: gaming brings developer awareness, robotics brings enterprise revenue.

### 8. Growth Loops Compound; Funnels Deplete

> "Content loops involve publishing and sharing media, which is then shared by the business, its users or partners. That action triggers signups, activations and user engagement, which leads to more sharing of the media."
> — Casey Winters, First Round Review ([firstround.com](https://review.firstround.com/pinterest-and-grubhubs-former-growth-lead-on-building-content-loops/))

World models have a natural content loop: user generates world, world gets a shareable URL (`marble.worldlabs.ai/world/{id}`), viewer sees the world, viewer wants to create their own. This is structurally identical to the Pinterest content loop that drove growth from 40 million to 200 million users. The GTM must instrument and optimize this loop — every generated world should be one click from shareable, every shared world should be one click from "create your own," and every creation should feed back into a public gallery that drives organic discovery through search.

### 9. Distribution Beats Product After Product-Market Fit

> "The general model for successful tech companies, contrary to myth and legend, is that they become distribution-centric rather than product-centric. They become a distribution channel, so they can get to the world."
> — Marc Andreessen, in Elad Gil's *High Growth Handbook* ([growth.eladgil.com](https://growth.eladgil.com/book/introduction/where-to-go-after-product-market-fit-an-interview-with-marc-andreessen/))

World Labs has achieved early product-market fit — the technology generates worlds that users find compelling. The next phase is distribution. This means the GTM must shift focus from "make the model better" to "get the model into more hands through more channels." The three surfaces (Marble App, World API, enterprise partnerships) are three distribution channels, each with different mechanics. The API is the most scalable because every developer who integrates it becomes a distribution node — their application puts World Labs technology in front of their users. Prioritize making the API easy to integrate over making the model incrementally better.

## Instructions

### Step 1: Define the Category

Produce a **Category Definition Document** that establishes "world models" as a recognized technology category.

1. Write a one-paragraph category definition that a CTO, a game designer, and a venture capitalist would all understand.
2. Identify the 3 closest existing categories and articulate how world models differ from each:
   - **3D generation tools** (e.g., Meshy, Tripo) — single objects vs. complete environments
   - **Video generation models** (e.g., Runway, Sora) — 2D pixels vs. persistent 3D geometry
   - **Game engines** (e.g., Unity, Unreal) — manual construction vs. AI generation
3. Define the "category story" arc: what was impossible before world models, what is now possible, and what becomes possible as the technology matures.
4. Identify the category's natural enemy (the status quo workflow it replaces) for each vertical.
5. Draft a category narrative that Fei-Fei Li's spatial intelligence framing and Lassner's Content 3.0 vision both reinforce.

**Output:** A 1-2 page Category Definition Document with positioning statements per audience.

### Step 2: Map the Multi-Surface GTM

Design a **GTM Surface Map** showing how the three product surfaces coordinate.

For each surface, define:

| Surface | Marble App (Consumer) | RTFM (Real-Time) | World API (Developer) | Enterprise Partnerships |
|---|---|---|---|---|
| **Primary metric** | MAU, worlds created/day | Sessions/day, avg session duration | API keys issued, monthly active integrations | Annual contract value, worlds generated/month |
| **Acquisition channel** | Social sharing, SEO, creator partnerships | Viral demo links, embeddable experiences | Documentation, GitHub, dev conferences | Direct sales, partner co-marketing, case studies |
| **Activation event** | First world generated | First interactive exploration | First successful API call | Signed POC agreement |
| **Retention mechanism** | Gallery, remix, community | Instant gratification, exploration depth | Reliability, new features, cost efficiency | Dedicated support, custom model tuning |
| **Revenue model** | Freemium + credits | Free preview → Marble export upsell | Pay-as-you-go credits | Volume commitments + support tiers |

Map the cross-surface flows:
- Marble App user discovers API through "build this into your app" CTA
- Developer discovers Marble App as a demo/showcase tool
- Enterprise prospect uses Marble App to evaluate quality before committing to API integration
- Enterprise partnership generates case study content that drives developer interest

**Output:** GTM Surface Map with acquisition flows, metrics, and cross-surface conversion paths.

### Step 3: Design the Developer GTM Engine

Produce a **Developer Growth Playbook** — the primary growth engine for the company.

1. **Documentation as the front door:** Design the docs architecture (quickstart, tutorials, API reference, cookbook, changelog). The first API call must be achievable in under 5 minutes. Map the documentation journey from "what is this?" to "I shipped it in production."
2. **Code as content:** Every blog post announcing a capability must ship with a working code example. Every partnership announcement must ship with an integration guide. Maintain a cookbook repository (modeled on OpenAI Cookbook / Anthropic Cookbook) with progressive-complexity recipes.
3. **Free tier design:** Recommend the optimal free credit grant that balances acquisition (enough to build something real) against abuse (not so much that it attracts pure hobbyists with no conversion intent). Benchmark against comparable APIs: OpenAI gives $5 free credit, Replicate gives $5, Stability gives 25 free credits.
4. **Developer relations calendar:** Map a 12-month cadence of hackathons, conference talks, livestreamed build sessions, and community spotlights.
5. **SDK investment strategy:** Prioritize Python SDK (robotics, ML engineers) > TypeScript/Node SDK (web developers, game developers) > Unity/Unreal plugins (game studios). Each SDK should wrap the full API surface including polling, asset download, and error handling.
6. **Build-in-public strategy:** Weekly changelog, public roadmap, "what we shipped this week" posts that build trust with developers who need to bet their stack on the platform.

**Output:** Developer Growth Playbook with documentation architecture, SDK roadmap, and 12-month devrel calendar.

### Step 4: Build the Vertical Penetration Plan

Produce a **Vertical Sequencing Matrix** that prioritizes market entry.

Score each vertical on five dimensions (1-5 scale):

| Dimension | Gaming & Immersive | Robotics & Simulation | Architecture & Design | Film & VFX |
|---|---|---|---|---|
| **Market readiness** | How ready are buyers to adopt? | | | |
| **Revenue per customer** | ACV potential | | | |
| **Volume per customer** | Worlds generated/month | | | |
| **Proof-of-concept speed** | Time from first meeting to POC | | | |
| **Ecosystem leverage** | Does winning here help win other verticals? | | | |

For each vertical, document:
- **Beachhead customer:** The specific partner or company type to land first (e.g., Escape.ai for film, Lightwheel for robotics)
- **Whole product gap:** What is missing between the current API and a complete solution for this vertical
- **Reference story:** The narrative arc of the case study you want to publish in 6 months
- **Competitive exposure:** Which competitors are strongest in this vertical

Recommend a sequencing order with rationale. A suggested starting framework:
1. **Gaming** (Phase 1) — highest developer volume, lowest sales cycle, strongest viral loop
2. **Robotics** (Phase 1-2) — highest revenue per customer, NVIDIA partnership as wedge, batch generation as killer use case
3. **Architecture** (Phase 2) — strong visual storytelling for case studies, emerging buyer awareness
4. **Film & VFX** (Phase 3) — longest sales cycle, highest production quality requirements, but marquee brand value

**Output:** Vertical Sequencing Matrix with beachhead customers, whole product gaps, and phased entry plan.

### Step 5: Design the Pricing and Packaging Strategy

Produce a **Pricing Architecture Document** with tier design and packaging rationale.

1. **Credit economics:** Map the cost structure. Plus model at ~1,500 credits/world (~$1.20/world). Mini model at ~150 credits/world (~$0.12/world). Text-to-world may incur pano generation + world generation (~2,000+ credits total for plus). Define the unit economics that make each tier viable.
2. **Tier design:**
   - **Free tier:** X credits/month, mini model only, watermarked outputs (or not — decide and justify)
   - **Pro tier:** $X/month, Y credits included, both models, full-resolution exports, API access
   - **Team tier:** $X/seat/month, shared credit pool, collaboration features, priority generation
   - **Enterprise tier:** Custom pricing, volume discounts, SLA, dedicated support, custom model fine-tuning
3. **Packaging decisions:** Should Marble App and World API share a credit pool or be separately priced? Should mesh export (currently plus-model only) be a feature gate or a model gate?
4. **Competitive pricing analysis:** How does this compare to Meshy ($20/mo for 200 credits), Runway ($12/mo for 625 credits), and Stability ($20/mo for 3,000 credits)?
5. **Enterprise volume incentives:** Design a volume discount curve that rewards commitment without cannibalizing self-serve revenue.

**Output:** Pricing Architecture Document with tier specifications, unit economics, and competitive benchmarks.

### Step 6: Build the Ecosystem Flywheel

Produce an **Ecosystem Flywheel Map** showing how open-source investment, developer tools, content creation, and user growth reinforce each other.

Define the flywheel stages:
1. **Open-source tooling** (renderer, examples) reduces friction for developers to display and work with generated worlds
2. **World API** gives those developers a way to generate content to display
3. **Generated worlds** are shared publicly, attracting viewers and new creators
4. **Cookbook / examples** lower the barrier for new developers to build integrations
5. **Third-party tools and plugins** (Unity, Unreal, Blender importers) expand the addressable surface
6. **Enterprise partnerships** generate marquee content that validates the technology for new verticals
7. **Community contributions** (recipes, tutorials, showcase projects) create organic discovery

For each flywheel stage, identify:
- The current state (working, nascent, or missing)
- The investment required to accelerate it
- The metric that proves it is working
- The bottleneck that would stall it

Map the reinforcing connections between stages (e.g., more cookbook recipes lead to more API integrations lead to more shared worlds lead to more Marble App signups lead to more community content).

**Output:** Ecosystem Flywheel Map with stage assessment, investment priorities, and reinforcement loops.

### Step 7: Create the Competitive Positioning Playbook

Produce a **Competitive Response Playbook** with pre-written positioning for each major competitor.

For each competitor, document:

**Google Genie 3:**
- Their frame: "Real-time explorable worlds from any image"
- Our counter-frame: "They have real-time. We have real-time AND persistence. Explore interactively with RTFM, then export production-quality 3D with Marble. They show you a world; we give you a world you can build on."
- Key proof points: RTFM matches their real-time capability PLUS API access, SPZ/GLB export, persistence, multi-modal input, Isaac Sim integration
- New advantage with RTFM: Genie 3's only edge (real-time) is neutralized. WL now spans both axes — real-time exploration AND exportable infrastructure.
- When this competitor comes up: sales calls with technical buyers, press comparisons, developer evaluations

**Runway GWM-1:**
- Their frame: "World model for video generation"
- Our counter-frame: "Runway generates pixels. We generate geometry. Their worlds exist as video frames; ours exist as persistent 3D environments you can walk through, export, and build on."
- Key proof points: 3D-native output, spatial consistency, exportable assets, interactive exploration

**NVIDIA Cosmos:**
- Their frame: "Open-source world foundation model for robotics and autonomous vehicles"
- Our counter-frame: This is not a competitor — it is a partner. Position as complementary. Cosmos provides the open-source base; World Labs provides the commercial, production-quality generation layer with full API access and multi-modal input.
- Strategic approach: Co-marketing, joint case studies, integration depth

**Decart / Odyssey / others:**
- General response framework for emerging competitors
- Positioning principles: lead with persistence, exportability, and developer access as the three differentiators that are hardest to replicate

For each competitor entry, include:
- One-line dismissal (for quick verbal responses)
- Detailed rebuttal (for written comparisons)
- Feature comparison table
- "Watch for" signals that this competitor is becoming a more serious threat

**Output:** Competitive Response Playbook with positioning scripts, proof points, and monitoring triggers.

## Diagnostic Questions

Before building the GTM plan, answer these questions to calibrate the strategy to the company's current state:

1. **Where is the primary revenue today?** If it is consumer credits (Marble App), the GTM should protect and grow that base. If it is enterprise contracts, the GTM should accelerate the sales pipeline. If revenue is negligible, the GTM should prioritize whichever surface has the shortest path to meaningful revenue.

2. **What is the current developer activation rate?** Of developers who create an API key, what percentage make a successful API call within 7 days? If this is below 40%, the developer GTM must prioritize onboarding friction reduction before any demand generation.

3. **Which vertical has the strongest existing pull?** Look at inbound inquiries, partnership conversations, and organic usage patterns. The GTM should go where the market is already pulling, not where the company wishes the market would pull.

4. **What is the generation quality gap between mini and plus?** If mini is "good enough" for most use cases, the pricing strategy should make mini the workhorse and reserve plus for premium tiers. If plus is significantly better, the pricing strategy should incentivize upgrades.

5. **How much of the developer experience is owned vs. outsourced?** If the Python SDK is maintained internally, it can be updated in lockstep with API changes. If community-maintained, the GTM must account for lag between API updates and SDK support.

6. **What is the company's appetite for a free tier?** A generous free tier accelerates developer adoption but requires infrastructure investment and fraud prevention. A restricted free tier protects margins but slows the flywheel. The right answer depends on whether the company is optimizing for growth or unit economics at this stage.

7. **Is the competitive environment stable or shifting?** If Google is about to open Genie 3 to developers, the GTM urgency is different than if the competitive landscape is static. Check for recent funding announcements, API launches, or partnership deals from competitors.

## Common Mistakes

### 1. Treating All Three Surfaces as One GTM

The Marble App, World API, and enterprise partnerships serve different buyers with different motivations, timelines, and purchasing processes. A single GTM plan that tries to address all three with the same messaging, channels, and metrics will underperform on every surface. Each surface needs its own positioning, its own funnel, and its own success metrics — coordinated by a shared category narrative but executed independently.

### 2. Leading with Technology Instead of Transformation

"Large World Models generate persistent 3D environments using Gaussian splatting" is a technology description, not a value proposition. Buyers do not care about Gaussian splatting. They care about generating a training environment in 30 seconds instead of 3 days. Every GTM asset must pass the "so what?" test: if the prospect cannot immediately see how this changes their workflow, the messaging has failed.

### 3. Pricing by Cost Instead of Value

Credit-based pricing is operationally convenient but strategically dangerous if the credit price is set by cost-plus logic rather than value capture. A robotics company that generates 500 training environments to improve robot pick-and-place accuracy by 15% is capturing enormous value. Pricing should reflect the value of the output (a trained robot that works better), not the cost of the input (GPU-seconds to generate a world).

### 4. Ignoring the Content Loop

Every generated world is a potential acquisition asset. If worlds are generated and never shared — because sharing is hard, or because shared worlds are not discoverable, or because viewers cannot easily create their own — the most powerful growth loop in the product goes unused. The GTM must treat the sharing and discovery infrastructure as a first-class product investment, not a nice-to-have social feature.

### 5. Sequencing Enterprise Before Developer

Enterprise deals are tempting because the contract values are large. But enterprise buyers in a new category want to see a thriving developer ecosystem before committing. They ask: "Who else is using this? Where are the case studies? Can I hire developers who know this API?" If the developer ecosystem is thin, enterprise deals stall. Invest in developer adoption first — it is the foundation that enterprise GTM is built on.

### 6. Treating NVIDIA as a Competitor Instead of a Force Multiplier

NVIDIA Cosmos is open-source. NVIDIA Isaac Sim is the dominant robotics simulation platform. Jensen Huang is publicly evangelizing digital twins and synthetic environments. This is not a competitive threat — it is the strongest possible channel partner. The GTM should maximize co-marketing, co-selling, and integration depth with NVIDIA, not waste energy on competitive positioning against Cosmos.
